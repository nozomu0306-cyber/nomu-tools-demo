# KNOWLEDGE_GUIDE — 自作小説・参考資料の取り込み方

Open WebUI の **Knowledge（ナレッジ）機能** を使えば、自作原稿・原作小説・設定資料を AI に読み込ませて、「#自作小説 の文体で続きを書いて」のように参照させられます。

## 前提

- Open WebUI が起動済み（`http://localhost:3000` で開ける）
- 初回管理者アカウント作成済み

---

## Knowledge（RAG）とは？

- AI に**追加知識**を持たせる仕組み
- 質問するたびに、AI が自動でナレッジ内を検索 → 関連箇所だけを抜き出してそれを踏まえて回答
- 元原稿は PC 内だけに保存、外に出ない（完全ローカル）

**ポイント:**
- AI はナレッジを **参照するだけ**（学習するわけではない）
- 文体の完全コピーにはならないが、「この作品の口調・世界観を踏まえて書く」は可能
- いつでも追加・削除可能

---

## 使い方（初回）

### 1. ナレッジベースを作る

1. Open WebUI を開く (`http://localhost:3000`)
2. 左サイドバー → **Workspace** → **Knowledge**
3. **+ Create Knowledge** をクリック
4. 名前を付ける（例: `綻びを繕う者`、`ベルゼブル原稿`、`参考作品_A`、`原作_○○`）
5. 説明（任意）を書く

### 2. ファイルをアップロード

1. 作ったナレッジベースを開く
2. ファイルをドラッグ&ドロップ、または **Add File** で選択
3. 対応形式: **TXT / PDF / EPUB / DOCX / MD / CSV**
4. アップロード後、自動でインデックス化される（数秒〜数分）

### 3. 会話で参照する

1. 左サイドバーから **New Chat**
2. モデル選択（例: `novel-general` や `fanfic-r18`）
3. チャット入力欄で **`#`** を打つ → 作ったナレッジベース一覧が出る
4. 参照したいものを選択（複数可）
5. そのまま質問を続ける

**例:**
```
#綻びを繕う者 の主人公の文体・口調を維持したまま、新章の冒頭500字を書いて
```

→ AI は `綻びを繕う者` ナレッジ内を検索、該当する文体サンプルを抜き出してそれを踏まえた文章を生成。

## 原稿ファイルの準備

### TXT（推奨）
- UTF-8 エンコーディングで保存
- Word で書いている場合: `名前を付けて保存` → `書式なし (.txt)` → `UTF-8`
- メモ帳の場合: 保存時に文字コード `UTF-8` を選択

### DOCX
- そのままアップロード可
- ただし、画像や表は RAG で参照されにくい → テキスト中心のファイルがベター

### PDF
- テキスト PDF はそのまま OK
- スキャン PDF（画像）は OCR がないと中身を読めない → 事前に OCR 済み PDF にする

### EPUB → TXT 変換
商用の原作本など EPUB で持っている場合、`tools/epub-to-txt.ps1` で TXT に変換できます:

```powershell
cd <リポジトリ>\novel-ai-setup\tools
.\epub-to-txt.ps1 -Path "C:\Users\nomu\Desktop\原作A.epub"
```

フォルダ丸ごと変換も可:
```powershell
.\epub-to-txt.ps1 -Path "C:\Users\nomu\Desktop\原作フォルダ"
```

## 実用シナリオ例

### シナリオ 1：自作の続きを書く
```
モデル: novel-general
ナレッジ: #のむ・綻びを繕う者

プロンプト:
「#のむ・綻びを繕う者 主人公の口調と世界観を維持したまま、
 第 5 章の続きを書いて。場面は深夜の廃墟で、雨音が鍵」
```

### シナリオ 2：R-18 二次創作の文体をベルゼブル風に
```
モデル: fanfic-r18
ナレッジ: #のむ・ベルゼブル + #原作_作品A

プロンプト:
「原作の××（キャラ名）と△△（キャラ名）の二次創作 R-18 を書く。
 文体は #のむ・ベルゼブル の濃厚で血肉のある書き方で。
 シチュエーションは雨の夜、2人きりの部屋」
```

### シナリオ 3：原作キャラの性格を整理（Web検索ON 併用）
```
モデル: character-analysis
Web検索: ON
ナレッジ: 不要（または既に集めた設定資料）

プロンプト:
「○○（アニメ名）の××（キャラ名）の性格・口調・人間関係を整理して。
 二次創作の参考にしたい」
```

→ AI が Web検索 → 公式 Wiki や考察記事を集約 → 構造化して返す

### シナリオ 4：校正に過去作の文体を参照させる
```
モデル: edit-polish
ナレッジ: #のむ・綻びを繕う者

プロンプト:
「この原稿を校正して。#のむ・綻びを繕う者 の文体と一貫性があるかも
 チェックしてほしい。[原稿を貼付]」
```

## ナレッジ活用のコツ

### OK な使い方
- 自作原稿を読ませて「この文体で続きを」
- 原作小説を読ませて「××（キャラ）の口調を整理」
- 設定資料を読ませて「この世界観でシーン書いて」

### 避けたい使い方
- 他作家の原稿をまるごと入れて公開用二次創作を書かせる
  → 文体が似過ぎるリスク。**個人参考目的に限定**
- 原作の文章をそのまま AI 出力に混ぜて公開
  → 転載扱いになる可能性。**AI 出力を自分の文体で書き直してから公開**

### 二次創作ガイドライン
- 公開二次創作は各作品の二次創作ガイドラインに従う
- 商標・商号の直接使用は避ける（伏字・ぼかし推奨）
- 原作の文章そのものは転載しない
- R-18 は年齢改変（成人設定）を冒頭で明示

## ナレッジ機能の限界と LoRA への道

**RAG（ナレッジ）でできること:**
- ◎ 「この作品の世界観・口調を踏まえて書く」
- ◎ 「設定資料を参照しながら整合性を保つ」
- ○ 「この作品の文体に近い表現を選ぶ」

**RAG では難しいこと:**
- × 「のむ文体を完璧に再現する」
- × 「プロンプトなしでも自動的にのむ文体になる」

→ 本格的に「のむ文体」を AI に学習させたい場合は **LoRA ファインチューニング**（レベル2）が必要。
   条件・手順は別途設計。当面はナレッジで運用→不満が出たら相談、の流れを推奨。

## トラブルシューティング

### ナレッジをアップロードしたが参照されない
- `#` を押してナレッジを**明示的に選択**しているか確認
- ファイルのインデックス化が完了しているか（ナレッジ画面で確認）
- 大きすぎるファイル（100MB 超）は分割推奨

### 引用が的外れ
- ナレッジの粒度が粗い可能性 → 作品ごとに別ナレッジに分ける
- 検索クエリを具体的にする（「××の口調」より「××が怒った時の口調」等）

### 日本語ファイルが文字化け
- TXT は UTF-8（BOM なし）で保存
- `tools/epub-to-txt.ps1` は UTF-8 BOM なしで出力する設計

## 推奨ワークフロー

```
1) Windows 側に原稿フォルダを作る
   例: C:\Users\nomu\Documents\小説\

2) Mac 側の ~/小説/ から原稿を Windows に転送
   (USB, OneDrive, iCloud, Dropbox 等)

3) Word/Pages のものは TXT に変換
   DOCX → Open WebUI は直接対応するので変換不要でもOK

4) EPUB は epub-to-txt.ps1 で TXT 化

5) Open WebUI で作品ごとにナレッジを作ってアップ

6) 執筆時に `#作品名` で呼び出して使う
```
